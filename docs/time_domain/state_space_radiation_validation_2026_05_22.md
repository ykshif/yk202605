# State-Space Radiation Validation - 2026-05-22

This note records the first validated state-space radiation approximation for
the external RODM time-domain adapter. The RODM frequency-domain solver remains
independent; the state-space work reads adapter-owned Cummins IRF outputs and
does not modify the RODM core frequency-domain workflow.

## Goal

The time-domain adapter now has two radiation-memory paths under investigation:

1. Direct Cummins convolution.
2. State-space radiation approximation.

The purpose of this step was not to replace the validated direct-convolution
baseline. It was to test whether the radiation memory kernel can be represented
by a compact state-space model that reproduces:

- the sampled Cummins radiation kernel;
- recovered frequency-domain added mass and radiation damping;
- the radiation-memory force generated by a known validated velocity history.

## Methods Implemented

### Common Real-Pole Approximation

The common-pole model fits the matrix radiation kernel as:

```text
K(t) ~= sum_i R_i exp(-p_i t)
```

This gives a simple continuous state-space realization:

```text
zdot_i = -p_i z_i + qdot
F_rad_memory = sum_i R_i z_i
```

This method is useful as a low-order baseline, but it cannot accurately
represent the oscillatory finite-band tail of the current radiation kernel.

### ERA Discrete State-Space Approximation

The ERA model fits the sampled explicit convolution weights:

```text
G_k = dt K(k dt), k >= 1
G_k ~= C A^(k-1) B
```

The zero-lag kernel is kept separately so the future solver can preserve the
same implicit zero-lag treatment used by the direct Cummins baseline.

## Closed-Loop Response Validation

An adapter-owned state-space Newmark solver was added:

```text

```

It recursively updates the ERA radiation state during the Newmark step, while
keeping the zero-lag radiation kernel in the same implicit damping treatment
used by the direct Cummins-convolution baseline.

Validation command:

```powershell
.\.venv\Scripts\python.exe scripts\validate_state_space_response.py `
  --state-order 240 `
  --era-block-rows 55 `
  --era-block-cols 55 `
  --output-root results\time_domain\state_space_response_dense88_era240_b55
```

Closed-loop response result:

```text
state_space_order = 240
kernel_fit_l2_relative_error = 6.521e-3
master_displacement_l2_relative_error = 6.976e-2
master_velocity_l2_relative_error = 4.543e-2
memory_force_l2_relative_error = 1.750e-2
memory_force_rms_relative_error = 8.086e-3
master_displacement_drift_slope_relative_error = 9.241e-2

centerline_heave_rms_relative_error = 2.041e-3
```

Important interpretation: ERA-240 reproduces the radiation-memory force well,
but the full motion response is more sensitive. The long random-wave case has
weakly constrained low-frequency drift; small state-space radiation differences
feed back through the motion equation and accumulate into a roughly 7% master
displacement L2 difference over 1209 s.

The DOF-by-DOF split shows that the vertical hydroelastic response is much
better than the full displacement norm suggests:

```text
master heave L2/RMS error = 6.764e-3 / 2.023e-3
master pitch L2/RMS error = 1.574e-2 / 2.463e-3
centerline heave L2/RMS error = 6.883e-3 / 2.041e-3
```

The large relative errors are concentrated in low-energy or drift-sensitive
components:

```text
surge L2 error = 5.582e-2, direct group RMS norm = 5.078e1
sway L2 error = 7.112e-1, direct group RMS norm = 3.002e0
roll L2 error = 6.106e-1, direct group RMS norm = 7.671e-7
```

This means the ERA state-space model is already strong for centerline heave and
pitch-like hydroelastic observables, while surge/sway drift still needs
mooring or low-frequency stabilization before full-motion production use.

Two checks clarify the issue:

```text
rectangular convolution-rule alignment:
  master displacement error = 7.617e-2
  memory-force error = 1.608e-2

ERA order 260:
  master displacement error = 1.588e-1
  memory-force error = 1.631e-2
  centerline heave error = 8.939e-3

ERA order 280:
  master displacement error = 1.466e-1
  memory-force error = 1.570e-2
  centerline heave error = 7.610e-3
```

So the remaining closed-loop error is not solved simply by switching the
convolution rule or increasing the ERA order. The next state-space step should
therefore focus on low-frequency/drift-sensitive validation, stability
constraints, and possibly balanced truncation or passivity-aware realization.

## Interpretation

The common real-pole model is not adequate as the production state-space
radiation model for the current dense-grid DM10 case. Increasing order without
regularization can become unstable, while regularized low-order fits remain too
inaccurate for memory-force reproduction.

The ERA model is much more promising. With enough order and Hankel block size,
it can closely reproduce the direct Cummins memory force. The ERA-240 result is
accurate enough to be treated as the first integration candidate, but it should
still be optimized before becoming the default production model.

After the closed-loop response validation, the conclusion is more nuanced:
ERA-240 is a good radiation-memory approximation and already reproduces the
centerline heave response very well. It is not yet a production replacement for
direct Cummins convolution for unconstrained full-body drift. It should remain
an optional adapter research path until mooring/low-frequency stabilization and
passivity-aware realization are added.

## Simple Mooring Check

A simple four-corner horizontal spring mooring was added to the adapter
validation path. The spring stiffness is assembled on the natural-order global
retained DOFs and projected to master coordinates with the SEREP transformation;
the RODM frequency-domain core is not modified.

Results for ERA-240:

| ---: | ---: | ---: | ---: | ---: |

| 1e6 N/m | 2.278e-2 | 1.758e-2 | 1.521e-2 | 6.889e-3 |
| 1e7 N/m | 6.874e-3 | 1.755e-2 | 1.891e-3 | 6.888e-3 |

## Architecture Boundary

This step preserved the intended architecture:

```text
RODM frequency-domain core
  -> exported hydrodynamic/frequency-domain data
  -> time_domain_adapter Cummins IRF
  -> time_domain_adapter state-space approximation
  -> validation diagnostics
```

No RODM frequency-domain core file was modified to depend on the state-space
adapter. The state-space code lives in `time_domain_adapter`, and the validation
script is independent.

## Next Step

The next practical step is to add an optional
`radiation_model = "state_space"` path to the adapter solver, but only after:

1. an ERA order sweep is used to choose a smaller stable production order;
2. the same validation is repeated for several peak periods / sea states;
3. the solver can load or receive a precomputed state-space model without
   rebuilding it inside the RODM frequency-domain workflow.
